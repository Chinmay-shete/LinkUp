Linkup - Technical & Project Report

1. Purpose of the Project

This project is a real-time web application inspired by Omegle. It allows users to connect with random strangers over the internet for anonymous text chatting and peer-to-peer video calling. The primary purpose is to provide a seamless, low-latency platform where users can instantly meet new people, with the ability to instantly skip ("Next") to a new person if they wish.

2. Technology Stack & Packages

  • Node.js & Express: The backend runtime and web framework used to serve the application and handle routing.
  • Socket.io: Enables real-time, bi-directional, and event-based communication. Used for the matchmaking queue, chat messages, and WebRTC signaling.
  • EJS (Embedded JavaScript templating): Used to render the HTML views dynamically on the server side.
  • Tailwind CSS: A utility-first CSS framework used for rapidly building the user interface without writing custom CSS.
  • WebRTC: Standard browser API used for peer-to-peer audio and video streaming. It allows direct media connections between two users for low latency.

3. User Flow & UX

1. Entry: When a user opens the app, they immediately enter the matchmaking queue. The UI displays "Looking for a stranger to chat with...".

2. Matchmaking: If another user is waiting, the server instantly pairs them up in a private "Room". The UI updates to allow texting.

3. Text Chatting: Users can exchange messages in real-time. Messages are styled distinctly (blue for self, gray for partner).

4. Video Calling: Either user can initiate a video call. The partner receives an incoming call prompt to Accept or Reject. Upon acceptance, WebRTC peer connections are established, and the video feeds appear on screen.

5. Skipping (The "Next" Button): If a user wants to end the conversation, they click Next. This immediately disconnects them, clears their chat, informs their partner, and drops them back into the matchmaking queue to find someone new instantly.

4. Core Logic & Architecture

Matchmaking (Queue System)

The backend maintains a waitingusers array. When a socket connects and requests a room, the server checks if this array has waiting users. If yes, it pairs the new user with the first user in the array by generating a unique room ID and subscribing both sockets to it. If empty, the user is pushed into the array.

WebRTC Signaling

Because WebRTC is peer-to-peer, browsers need a way to exchange connection data (SDP offers/answers and ICE candidates) before establishing a direct link. The socket.io server acts as a signaling server, blindly relaying signalingMessage events between the two users in a room until their direct video connection is established.

Room Security

All events (messages, video calls, hanging up) are strictly validated on the backend. The server checks that the socket emitting the event actually belongs to the specified room, preventing malicious users from hijacking other rooms.
